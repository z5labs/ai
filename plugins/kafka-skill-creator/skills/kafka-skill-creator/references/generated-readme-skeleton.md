# Generated-skill README skeleton

The README that ships inside every generated `kafka-<team>/`. Read this file when executing Step 3 of `SKILL.md` (specifically the `<output>/README.md` substep). The high-level reasoning lives in `SKILL.md`; this file carries the verbatim template.

Substitute every `<...>` placeholder with real values from the manifest at generation time so the examples are runnable copy-paste. In particular:

- `<team>`, `<env list>` — straight from the manifest
- `<bullet list>` for owned topics and groups — emit one `- ` line per item
- `<first-topic>` and `<first-group>` — pick the first entry from `topics:` and `consumer_groups:` respectively (deterministic so the README's examples work for the most-recently-generated skill)

```markdown
# kafka-<team>

Read-only investigation of the **<team>** team's Kafka topics and consumer groups across <env list>.

Generated by `/kafka-skill-creator` from `scripts/manifest.yml`. Do not hand-edit files in this directory — re-running the generator overwrites everything.

## Posture

Non-destructive only. The wrappers do reads (`consume`, `describe-topic`, `describe-group`, `lag`) and offset resets (`reset-offsets`, with `--dry-run`). They do **not** produce messages, create / alter / delete topics, delete consumer groups, or change ACLs. To do any of those, use `kafkactl` directly with your own credentials — that's deliberately out-of-band so it doesn't ride on the team's automation.

## What's owned

- Topics: <bullet list>
- Consumer groups: <bullet list>

The wrappers refuse any topic or group not in this list. Edit `scripts/manifest.yml` and re-run `/kafka-skill-creator --manifest …` to extend it.

## One-time setup

\`\`\`bash
# Per environment the team operates against:
cp scripts/.env.example .env.dev
# edit .env.dev — fill in CONTEXTS_DEV_BROKERS / _SASL_USERNAME / _SASL_PASSWORD
# (and CONTEXTS_DEV_SCHEMAREGISTRY_* if Schema Registry is configured)

# Add the env files to .gitignore — they contain secrets:
echo '/.env' >> .gitignore
echo '/.env.*' >> .gitignore
\`\`\`

For locked-down environments, populate the env vars via a credential helper instead of a plain `.env` file:

\`\`\`bash
op run --env-file=kafka.env -- claude
\`\`\`

…where `kafka.env` declares e.g. `CONTEXTS_DEV_SASL_PASSWORD=op://Vault/Kafka/dev-password`.

## Sample uses

All examples assume the wrappers are at `scripts/` (relative to this README). When invoked from elsewhere, use the full path (`./.claude/skills/kafka-<team>/scripts/...` or `plugins/team-<team>/skills/kafka-<team>/scripts/...`).

### Peek at recent messages on a topic

\`\`\`bash
# Last few messages from prod, with Schema Registry deserialization (if configured):
bash scripts/consume.sh <first-topic> --context prod --max 5

# Tail from the beginning of the partition (good for low-traffic topics):
bash scripts/consume.sh <first-topic> --context staging --from-beginning --max 20

# Pin to a specific partition:
bash scripts/consume.sh <first-topic> --context dev --partition 3 --max 10
\`\`\`

### Describe a topic's config and partitions

\`\`\`bash
bash scripts/describe-topic.sh <first-topic> --context prod
\`\`\`

Output is JSON — pipe through `jq` to extract specific fields:

\`\`\`bash
bash scripts/describe-topic.sh <first-topic> --context prod | jq '.partitions | length'
\`\`\`

### Check consumer-group lag

\`\`\`bash
# Quick lag-only view:
bash scripts/lag.sh <first-group> --context prod

# Full describe (members, subscriptions, lag in one):
bash scripts/describe-group.sh <first-group> --context prod
\`\`\`

### Reset offsets (with dry-run)

\`\`\`bash
# Always start with --dry-run; review the planned offsets before omitting it:
bash scripts/reset-offsets.sh <first-group> \
  --topic <first-topic> \
  --to-earliest \
  --context staging \
  --dry-run

# Reset to a specific offset (per-partition, kafkactl applies to all assigned partitions):
bash scripts/reset-offsets.sh <first-group> \
  --topic <first-topic> \
  --to-offset 1234 \
  --context staging \
  --dry-run
\`\`\`

`reset-offsets.sh` honors kafkactl's built-in refusal to reset against a consumer group with active members — there is no `--force` or `--allow-active-members` flag on the wrapper. Stop the consumer first if you really need to reset against a live group.

## Selecting an environment

The wrappers resolve a `.env` file in this order, first match wins:

1. `--env-file PATH`
2. `KAFKA_ENV_FILE` env var
3. `./.env`

\`\`\`bash
# Explicit per-invocation:
bash scripts/lag.sh <first-group> --context prod --env-file .env.prod

# Whole-shell:
export KAFKA_ENV_FILE=.env.staging
bash scripts/describe-topic.sh <first-topic> --context staging
\`\`\`

A loaded `.env` that doesn't populate the chosen `--context` fails closed: kafkactl looks up `CONTEXTS_<CTX>_*` for the context you passed, and if those vars aren't set, the call exits with a missing-creds error before reaching the broker.

## Regeneration

Topics drift, groups come and go, schemas change. To pull in updates, edit `scripts/manifest.yml` and re-run the generator:

\`\`\`bash
/kafka-skill-creator --manifest scripts/manifest.yml --output <this-directory>
\`\`\`

This **overwrites** every file in this directory. Treat any local edits to generated files as ephemeral.

## Where things live

- `SKILL.md` — what Claude reads when this skill triggers; not meant for human reference.
- `scripts/` — the wrappers, plus `manifest.yml` (the team's source of truth) and `.env.example`.
- `references/` — generation-time snapshots of cluster metadata, topic configs, group state, and Schema Registry schemas.
```
