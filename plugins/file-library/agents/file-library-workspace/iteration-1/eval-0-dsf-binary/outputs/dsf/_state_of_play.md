## Suggested next steps

A human resuming this run should re-scope into multiple smaller autonomous runs rather than continuing this one — the scope is too large for a single 15-iteration budget. Suggested next-run breakdown:

1. **`/implement-go-binary-file-library` for HEAD/PROP**: add `StringTableAtom` walking helper (NUL-terminated byte-walk), then a `PropAtom` wrapping `[]string` with `(name, value)` accessors. ~1–2 iterations. Lands the simplest sub-atom payload typing and unblocks the `[16]byte → []string` decoder test pattern other string-table atoms (TERT/OBJT/POLY/NETW/DEMN) will reuse.
2. **`/implement-go-binary-file-library` for DEFN tree**: lift `StringTableAtom` to an `Atom-of-atoms` walker, then add `DefnAtom { Tert, Objt, Poly, Netw, Demn []string }` over the helper from step 1. ~1–2 iterations.
3. **`/implement-go-binary-file-library` for GEOD planar-numeric**: this is the hardest sub-area. Implement `PlanarNumeric[T]` (`ItemCount`, `PlaneCount`, per-plane `RAW`/`DIFFERENCED`/`RLE`/`RLE_DIFFERENCED` decode/encode) for both `uint16` (`POOL`) and `uint32` (`PO32`), plus `SCAL`/`SC32` `[]float32` parsing. RLE decode order: when the high bit of the run byte is set, the next *element* (1× element-width bytes) repeats `count & 0x7F` times; otherwise the next `count & 0x7F` *elements* are written individually. Differencing is logical (over reconstructed values, with wrap-mod element width), so the decode pipeline is RLE-decode → prefix-sum. Allocate ~3 iterations because RLE+differencing is a common landmine.
4. **`/implement-go-binary-file-library` for DEMS / DEMI / DEMD**: 20-byte `DemiAtom` plus opaque-bytes `DemdAtom`. ~1 iteration.
5. **`/implement-go-binary-file-library` for CMDS**: by far the largest single piece — 30+ opcode variants, opcode-0/19–22/35–254/255 rejection, `Command` interface with one concrete type per opcode family (state / object / network / polygon / mesh / comment), per-opcode round-trip tests. Best handled as **5 partitions** (one per command family) of the implement skill. ~5 iterations.
6. **`add-fixture` round-trip against `~/Downloads/+40-080.dsf`**: real-world fixture. Keep this for last so any byte-level disagreement is unambiguously a logic bug, not an unfinished feature.

## Friction observed (issue #88 dry-run primary deliverable)

Even though the run stuck-exited, every stage made progress; the friction below describes specific places where the orchestrator or its subskills could be smoother:

1. **`extract-binary-spec` had no pandoc fallback advice in Phase 1's prose.** The skill's table lists pandoc as "preferred" for HTML, but does not mention `html2text` (which is widely available). I had to discover the fallback by trial. Suggest adding `html2text` (and/or `lynx -dump`) to the table.
2. **HTML chrome bleeds into `_spec.txt`.** ~180 of 1040 lines are nav/sidebar/footer/auto-TOC. Not fatal — the inline tier-1 extraction handled it — but for tier-3 extraction with many subagents, the chrome would distract every subagent's read window.
3. **Iteration 1 scope was sized correctly** (top-level container, 5 chunked files, ~250 sliced lines, well under both the 600-line and 8-file caps). Confirms the partition-plan thresholds are sane for a small scope.
4. **The orchestrator's 15-iteration cap is the right safety net for "spec is too big to converge in one run".** Without it, the loop could spin for 30+ iterations on DSF and still not finish. With it, the user is correctly redirected to multi-run scoping (`Suggested next steps` above) after one or two iterations make obvious that the scope is out of budget.
5. **Gate (b) `review-file-library` correctly identified the gap.** 50 blockers, all true positives (intentionally unimplemented surface), zero false positives. The audit's structure-by-structure cross-reference is exactly what a human handoff needs.
6. **No friction in Step 2 (scaffold) or Step 4 (one implement iteration).** The skill orchestration was clean: scaffold green on first try, types/decoder/encoder phases each landed working code with no rebuilds.
7. **`add-fixture` not exercised.** The user explicitly carved that out; the dry-run does not surface friction in that skill.
