# Generated-skill README skeleton

The README that ships inside every generated `pg-<dbname>/`. Read this file when executing Step 2 of `SKILL.md` (specifically the `<output>/README.md` substep). The high-level reasoning lives in `SKILL.md`; this file carries the verbatim template.

Substitute every `<...>` placeholder with real values from introspection at generation time so the examples are runnable copy-paste. In particular:

- `<dbname>` — the value of `PGDATABASE` (also the leaf segment of the output path, `pg-<dbname>/`)
- `<top tables>` — the same 3–5 prominent table names you put in the SKILL.md frontmatter description (highest column count or most-referenced by FKs)
- `<top-table>` — the single most prominent table from `<top tables>`, used in the sample-use code blocks so the count example is runnable copy-paste
- `<table count>`, `<view count>`, `<enum count>` — totals from the introspection output (drop the enums clause entirely if `enums.tsv` was empty)

## Posture

This skill is intended for an analyst or developer doing ad-hoc reads, one-off DML, and manual exploration. The wrapper (`scripts/query.sh`) runs whatever SQL you give it — it is not restricted to reads — so for destructive statements the SKILL.md instructions ask Claude to dry-run via `BEGIN; ... ROLLBACK;` and surface the affected rows before executing for real. Treat this skill the way you'd treat a `psql` session against the same database.

## One-time setup

Run these commands from your project root (the directory containing `.claude/`) so the `.env` files and `.gitignore` rule land outside the generated skill directory:

\`\`\`bash
# Per environment you connect to:
cp .claude/skills/pg-<dbname>/scripts/.env.example .env.dev
# edit .env.dev — fill in PGHOST, PGPORT, PGUSER, PGPASSWORD
# (PGDATABASE is intentionally absent — the dbname is hardcoded in query.sh)

# Add the env files to .gitignore — they contain secrets. The `grep -qxF`
# guard makes this re-runnable: appending only when the exact line is not
# already present, so following this README twice doesn't duplicate rules.
touch .gitignore
grep -qxF '/.env'   .gitignore || echo '/.env'   >> .gitignore
grep -qxF '/.env.*' .gitignore || echo '/.env.*' >> .gitignore
\`\`\`

For locked-down environments, populate the env vars via a credential helper instead of a plain `.env` file:

\`\`\`bash
op run --env-file=postgres.env -- claude
\`\`\`

…where `postgres.env` declares e.g. `PGPASSWORD=op://Vault/Postgres/dev-password`. The same shape works with `vault`, `direnv`, `gcloud`, or any helper that exports secrets to a subprocess.

## Sample uses

All commands below assume your shell is at the project root, so the wrapper is invoked by its full path (`.claude/skills/pg-<dbname>/scripts/query.sh`). The `query.sh` resolution order picks up `./.env` from the same project root.

### Smoke test

\`\`\`bash
bash .claude/skills/pg-<dbname>/scripts/query.sh "SELECT 1"
\`\`\`

If this fails, the wrapper or the `.env` is misconfigured — fix it before running anything else.

### Quick row count from a real table

\`\`\`bash
bash .claude/skills/pg-<dbname>/scripts/query.sh "SELECT count(*) FROM <top-table>"
\`\`\`

### Multi-statement scripts via stdin

\`\`\`bash
bash .claude/skills/pg-<dbname>/scripts/query.sh < script.sql
\`\`\`

Useful for migrations-style work or anything that needs more than one statement (the `-c` form `psql` runs by default rejects multi-statement input in some contexts).

### Per-environment

\`\`\`bash
bash .claude/skills/pg-<dbname>/scripts/query.sh --env-file .env.prod "SELECT count(*) FROM <top-table>"
\`\`\`

Or set `PG_ENV_FILE` once per shell:

\`\`\`bash
export PG_ENV_FILE=.env.staging
bash .claude/skills/pg-<dbname>/scripts/query.sh "SELECT now()"
\`\`\`

Resolution order is `--env-file` → `PG_ENV_FILE` → `./.env`, first match wins. If `--env-file` or `PG_ENV_FILE` points at a missing file, the wrapper exits with an error rather than silently falling through to defaults — a typo shouldn't quietly send you to the wrong environment.

## Schema lookup

Before writing a non-trivial query, consult the relevant reference under `references/`:

- `references/tables.md` — every table's columns, types, nullability, defaults, PK/FK annotations
- `references/relationships.md` — flat list of foreign keys in arrow form (`orders.user_id → users.id`)
- `references/views.md` — view definitions verbatim (often pre-joined query patterns worth reusing)
- `references/indexes.md` — one line per index, schema-qualified, for judging WHERE-clause cost
- `references/enums.md` — enum types and their values (only present if the database defines any)

## Regeneration

Schemas drift. To pull in changes, re-export the same libpq env vars and re-run the generator:

\`\`\`bash
/postgres-skill-creator
\`\`\`

This **overwrites** every file under `pg-<dbname>/` — keep any project-specific guidance somewhere else (top-level `CLAUDE.md`, a sibling skill, etc.) so it survives regeneration.

## Where things live

- `SKILL.md` — what Claude reads when this skill triggers; not meant for human reference.
- `scripts/query.sh` — the `psql`-in-a-container wrapper; dbname baked in, host/port/user/password from `.env`.
- `scripts/.env.example` — the keys `query.sh` understands, with the generation-time values pre-filled as commented defaults.
- `references/` — generation-time snapshots of tables, FKs, views, indexes, and enums.
