# Generated-skill README skeleton

The README that ships inside every generated `pg-<dbname>/`. Read this file when executing Step 2 of `SKILL.md` (specifically the `<output>/README.md` substep). The high-level reasoning lives in `SKILL.md`; this file carries the verbatim template.

Substitute every `<...>` placeholder with real values from introspection at generation time so the examples are runnable copy-paste. In particular:

- `<dbname>` ‚Äî the value of `PGDATABASE` (also the leaf segment of the output path, `pg-<dbname>/`)
- `<top tables>` ‚Äî the same 3‚Äì5 ranked tables you used in the SKILL.md frontmatter description, computed by the deterministic **Top-table ranking** rule defined in SKILL.md (FK in-degree DESC ‚Üí column count DESC ‚Üí `(schema, table_name)` ASC). Render each as a bare `table` name ‚Äî this is prose for humans, not SQL.
- `<top-table>` ‚Äî the **first** entry of `<top tables>`, formatted for SQL as a **schema-qualified, double-quoted identifier**: `"<schema>"."<table>"`. The README's row-count sample is supposed to be runnable copy-paste, so a bare `users` won't do ‚Äî it would fail on `analytics."UserSessions"`-shaped tables, on duplicate names across schemas, and on identifiers that need quoting (mixed case, spaces, reserved words). Always-quote is safe because Postgres treats `"users"` and `users` as the same object when the stored name is lowercase.
- `<table count>`, `<view count>`, `<enum count>` ‚Äî totals from the introspection output

When `enums.tsv` is empty, the generator does not write `references/enums.md`. The README must not advertise content that doesn't exist, so in that case strip every enum mention from the template before writing ‚Äî there are three of them:

1. The schema-summary line: drop `, and <enum count> enums` so it reads `‚Ä¶ describe <table count> tables, <view count> views (top tables: ‚Ä¶)`.
2. The Schema-lookup section: drop the `references/enums.md` bullet entirely.
3. The Where-things-live section: drop `, and enums` from the `references/` bullet so it reads `‚Ä¶ snapshots of tables, FKs, views, and indexes.`

If `enums.tsv` is non-empty, leave all three places as written and substitute `<enum count>` with the integer.

# pg-<dbname>

Ad-hoc SQL access to the **<dbname>** Postgres database. Schema captured by `/postgres-skill-creator` from the live database; reference docs under `references/` describe <table count> tables, <view count> views, and <enum count> enums (top tables: <top tables>).

Generated by `/postgres-skill-creator`. Do not hand-edit files in this directory ‚Äî re-running the generator overwrites everything under `pg-<dbname>/`. Keep the `.env` files and `.gitignore` rules below at the project root (not inside this directory) so regeneration leaves them alone.

## Posture

This skill is intended for an analyst or developer doing ad-hoc reads, one-off DML, and manual exploration. The wrapper (`scripts/query.sh`) runs whatever SQL you give it ‚Äî it is not restricted to reads ‚Äî so for destructive statements the SKILL.md instructions ask Claude to dry-run via `BEGIN; ... ROLLBACK;` and surface the affected rows before executing for real. Treat this skill the way you'd treat a `psql` session against the same database.

## One-time setup

Run these commands from your project root (the directory containing `.claude/`) so the `.env` files and `.gitignore` rule land outside the generated skill directory:

\`\`\`bash
# Default environment ‚Äî query.sh picks ./.env up automatically when no
# --env-file flag is passed, so the smoke test below is runnable as-is
# once this file is filled in.
cp .claude/skills/pg-<dbname>/scripts/.env.example .env
# edit .env ‚Äî fill in PGHOST, PGPORT, PGUSER, PGPASSWORD
# (PGDATABASE is intentionally absent ‚Äî the dbname is hardcoded in query.sh)

# (Optional) per additional environment, referenced explicitly via
# --env-file or PG_ENV_FILE in the "Per-environment" section below:
#   cp .claude/skills/pg-<dbname>/scripts/.env.example .env.prod

# Add the env files to .gitignore ‚Äî they contain secrets. The `grep -qxF`
# guard makes this re-runnable: appending only when the exact line is not
# already present, so following this README twice doesn't duplicate rules.
# `/.env.*` covers .env.prod, .env.staging, etc.
touch .gitignore
grep -qxF '/.env'   .gitignore || echo '/.env'   >> .gitignore
grep -qxF '/.env.*' .gitignore || echo '/.env.*' >> .gitignore
\`\`\`

For locked-down environments, populate the env vars via a credential helper instead of a plain `.env` file:

\`\`\`bash
op run --env-file=postgres.env -- claude
\`\`\`

‚Ä¶where `postgres.env` declares e.g. `PGPASSWORD=op://Vault/Postgres/dev-password`. The same shape works with `vault`, `direnv`, `gcloud`, or any helper that exports secrets to a subprocess.

## Sample uses

All commands below assume your shell is at the project root, so the wrapper is invoked by its full path (`.claude/skills/pg-<dbname>/scripts/query.sh`). The `query.sh` resolution order picks up `./.env` from the same project root.

### Smoke test

\`\`\`bash
bash .claude/skills/pg-<dbname>/scripts/query.sh "SELECT 1"
\`\`\`

If this fails, the wrapper or the `.env` is misconfigured ‚Äî fix it before running anything else.

### Quick row count from a real table

\`\`\`bash
bash .claude/skills/pg-<dbname>/scripts/query.sh "SELECT count(*) FROM <top-table>"
\`\`\`

### Multi-statement scripts via stdin

\`\`\`bash
bash .claude/skills/pg-<dbname>/scripts/query.sh < script.sql
\`\`\`

Useful for `.sql` files you already have on disk, multi-line SQL that's awkward to inline as a single shell argument, and SQL that mixes plain statements with psql meta-commands (`\copy`, `\set`, `\timing`, `\d`, ‚Ä¶). Plain multi-statement SQL works fine via the inline `-c` form too ‚Äî `bash .claude/skills/pg-<dbname>/scripts/query.sh "SELECT 1; SELECT 2"` runs both statements in a single transaction ‚Äî but `-c` can't mix SQL with psql meta-commands, so anything using `\copy` or similar has to come in via stdin.

### Per-environment

\`\`\`bash
bash .claude/skills/pg-<dbname>/scripts/query.sh --env-file .env.prod "SELECT count(*) FROM <top-table>"
\`\`\`

Or set `PG_ENV_FILE` once per shell:

\`\`\`bash
export PG_ENV_FILE=.env.staging
bash .claude/skills/pg-<dbname>/scripts/query.sh "SELECT now()"
\`\`\`

Resolution order is `--env-file` ‚Üí `PG_ENV_FILE` ‚Üí `./.env`, first match wins. If `--env-file` or `PG_ENV_FILE` points at a missing file, the wrapper exits with an error rather than silently falling through to defaults ‚Äî a typo shouldn't quietly send you to the wrong environment.

## Schema lookup

Before writing a non-trivial query, consult the relevant reference under `references/`:

- `references/tables.md` ‚Äî every table's columns, types, nullability, defaults, PK/FK annotations
- `references/relationships.md` ‚Äî flat list of foreign keys in arrow form (`orders.user_id ‚Üí users.id`)
- `references/views.md` ‚Äî view definitions verbatim (often pre-joined query patterns worth reusing)
- `references/indexes.md` ‚Äî one line per index, schema-qualified, for judging WHERE-clause cost
- `references/enums.md` ‚Äî enum types and their values

## Regeneration

Schemas drift. To pull in changes, re-export the same libpq env vars and re-run the generator:

\`\`\`bash
/postgres-skill-creator
\`\`\`

This **overwrites** every file under `pg-<dbname>/` ‚Äî keep any project-specific guidance somewhere else (top-level `CLAUDE.md`, a sibling skill, etc.) so it survives regeneration.

## Where things live

- `SKILL.md` ‚Äî what Claude reads when this skill triggers; not meant for human reference.
- `scripts/query.sh` ‚Äî the `psql`-in-a-container wrapper; dbname baked in, host/port/user/password from `.env`.
- `scripts/.env.example` ‚Äî the keys `query.sh` understands, with the generation-time values pre-filled as commented defaults.
- `references/` ‚Äî generation-time snapshots of tables, FKs, views, indexes, and enums.
