## Summary

The skill is small and largely well-shaped: clear single responsibility, idempotent contract, sensible "when to skip" guidance, and an `argument-hint` that matches the inputs section. There is **one correctness bug** that would cause the produced report to mislabel two of the three numbers, and a handful of robustness gaps around input validation, quoting/path handling, and the contract between flag order and output order. None of the issues are stylistic — they all change observable behavior.

Severity legend: **High** = wrong output or data-loss risk; **Medium** = predictable failure on reasonable inputs; **Low** = polish / clarity.

---

## Findings

### H1. `wc -wlc` field order does not match the order the skill claims (High — produces wrong report)

`SKILL.md` line 24:
> Run `wc -wlc "$path"` and capture the three numbers from the first whitespace-delimited fields.

And line 19 documents the report as `words: N`, `lines: N`, `bytes: N` in that order.

**Problem:** GNU `wc` ignores the flag order and always prints counts in a fixed canonical order: **newlines, words, bytes (then chars, max-line-length)**. Verified locally:

```
$ printf "hello world\nsecond line\n" > /tmp/x ; wc -wlc /tmp/x
 2  4 24 /tmp/x        # lines=2, words=4, bytes=24
```

So "first whitespace-delimited field" is the **line** count, not the word count. A naive implementer following the workflow verbatim will write `words: 2`, `lines: 4`, `bytes: 24` — silently swapping words and lines. This is the kind of bug that survives review because the numbers all look plausible.

**Fix options (pick one and make the workflow unambiguous):**
- Call `wc` three times with single flags: `wc -w "$path"`, `wc -l "$path"`, `wc -c "$path"`. Cheapest in clarity, three forks instead of one.
- Keep one `wc -wlc` call but document the actual order: "fields are lines, words, bytes — assign accordingly."
- Use named extraction, e.g. `read lines words bytes _ < <(wc "$path")`, with the variable names matching `wc`'s real output.

### H2. Output path collision with the input file is not handled (High — possible data loss on re-run)

The skill writes `<path>.count.md`. If the input is itself `notes.count.md` (a previous run's output, or any file with that name), the new output is `notes.count.md.count.md` — fine. But:

- If the user passes `notes.md` and a file `notes.md.count.md` already exists from an earlier hand-edit (with notes the user wanted to keep), step 3 silently overwrites it. The skill calls overwrite "the documented refresh path," which is reasonable for *its own* prior outputs but does not distinguish them from a user-authored file at the same path.
- There is no marker (header line, magic comment) that lets a future run recognize "this is a word-count report I previously wrote, safe to overwrite" vs. "this is a user file, refuse."

**Fix:** Either (a) write a recognizable header line (e.g. `<!-- generated by word-count skill -->`) and refuse to overwrite a destination that lacks it, or (b) explicitly call out in the doc that *any* file at `<path>.count.md` will be clobbered, so the user can choose the input path knowingly.

### M1. `text/` mime prefix rejects real text formats (Medium — false negatives on common inputs)

Line 15:
> validate ... must be a text file (test with `file --mime-type`; reject if it doesn't start with `text/`).

Verified mime types from `file --mime-type`:

| File             | mime type            | accepted by `text/` rule? |
| ---------------- | -------------------- | ------------------------- |
| empty `.txt`     | `inode/x-empty`      | no                        |
| `.json`          | `application/json`   | no                        |
| `.svg`           | `image/svg+xml`      | no                        |
| `.csv` (simple)  | `text/plain`         | yes                       |

So an **empty file** — which has perfectly well-defined word/line/byte counts (0/0/0) — is rejected, and so are JSON and SVG, which the user almost certainly considers text. The skill description ("text file") is broader than the validation rule.

**Fix:** Either widen the predicate (accept `text/*`, `application/json`, `application/xml`, `inode/x-empty`, and friends) or narrow the description ("plain-text file as detected by `file --mime-type text/...`") so the contract matches the check.

### M2. Path quoting / argument handling is underspecified (Medium)

The workflow shows `wc -wlc "$path"` (good — quoted), but:

- It does not say to use `--` to terminate options. A path beginning with `-` (e.g. `-rf.txt`) will be interpreted as a flag by `wc`, `file`, and `[ -f ]`. Use `wc -wlc -- "$path"` and `file --mime-type -- "$path"`.
- It does not say what to do when `path` is a symlink. `[ -f "$path" ]` follows symlinks; `wc` follows symlinks; `file --mime-type` follows symlinks by default. That's probably the desired behavior, but the skill should state it (or use `[ -f "$path" ] && [ ! -L "$path" ]` if it should refuse symlinks).
- It does not mention permission errors (file exists, mime ok, but unreadable). Currently `wc` would fail and the report would not be written, but the user-facing error path is undefined.

### M3. "Sibling" output assumes the input directory is writable (Medium)

Line 9 / line 19 specify that the report goes to `<path>.count.md` next to the input. If `path` is in a read-only directory (e.g. `/usr/share/...`, a mounted snapshot, a Docker image rootfs), the write fails after validation succeeds. The skill should either:

- explicitly require the parent directory be writable in step 1, **or**
- support an output-directory override.

### L1. `argument-hint` says one path; description says "a specific file or set of files" (Low — minor mismatch)

Frontmatter `argument-hint: "<path-to-file>"` (singular) and Inputs ("a regular file", singular) are consistent. But the **description** field says:

> Use when the user asks for word/line/byte counts on a specific file **or set of files**.

The skill has no loop, no glob handling, and no aggregation. Either remove "or set of files" from the description, or extend the workflow to accept multiple paths and write one report per file. Currently this is a triggering-vs-capability mismatch that will pull the skill in for requests it can't fulfill in one invocation.

### L2. "Three-number report" vs. report format is slightly underspecified (Low)

Line 19 says the report "contains three lines: `words: N`, `lines: N`, `bytes: N`." Trailing newline? Header? Timestamp? File path of the source? None of these are required, but pinning them down (or saying "exactly these three lines, no header, trailing newline") makes the idempotency guarantee testable. Right now two implementers writing this skill would produce byte-different reports and both could claim conformance.

### L3. No mention of running on Windows / non-GNU `wc` (Low)

`wc` field order is consistent across BSD and GNU, but `file --mime-type` differs (BSD `file` on macOS prints slightly different strings, e.g. `regular file` for unrecognized). If the skill is expected to run cross-platform, note it; if Linux/GNU only, say so.

## Things the skill does well

- Single, clear responsibility; "When to skip" lists realistic adjacent requests and points away from this skill instead of trying to grow.
- Idempotency is named and explained, not just implied.
- `argument-hint` matches the actual input.
- The validation step is in the workflow before any write, so a bad input won't produce a partial report.
- The description's positive trigger ("word/line/byte counts on a specific file") is concrete enough to discriminate against the negative triggers in the same sentence.

## Recommended minimum patch

If only one change ships, fix **H1**: change step 2 to call `wc` three times (or document the real `lines, words, bytes` field order from `wc -wlc`). Everything else is robustness and contract-tightening.

A reasonable medium-effort patch would also:

1. Replace the `text/` prefix check with an allowlist that includes `inode/x-empty` and `application/json` (M1).
2. Add `--` before `"$path"` in every command shown (M2).
3. Drop "or set of files" from the description **or** make the workflow loop (L1).
